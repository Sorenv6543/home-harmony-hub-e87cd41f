# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Claro** — a short-term rental cleaning & booking management dashboard (property management, booking timeline, recurring cleaning schedules, calendar sync). Built with TanStack Start and scaffolded/synced by the **Lovable.dev** platform, which owns and regenerates certain files (see "Generated files" below).

## Commands

Package manager: `bun` (`bunfig.toml` present; `bun.lock` is the primary lockfile). A `package-lock.json` also exists — prefer `bun` unless told otherwise.

```bash
bun install       # install deps
bun run dev       # vite dev server
bun run build     # production build (vite build)
bun run build:dev # development-mode build
bun run preview   # preview a production build
bun run lint      # eslint .
bun run format    # prettier --write .
```

There is no test suite in this repo currently.

## Generated files — do not hand-edit

Several files carry `// This file is automatically generated` or `// AUTO-GENERATED by @lovable.dev/mcp-js` banners. Treat these as owned by tooling:

- `src/integrations/supabase/*` (`client.ts`, `client.server.ts`, `auth-attacher.ts`, `auth-middleware.ts`, `types.ts`) — regenerated by Lovable's Supabase integration.
- `src/integrations/lovable/index.ts` — Lovable auth glue.
- `src/routes/mcp.ts`, `src/routes/[.well-known]/oauth-protected-resource.ts`, `src/routes/[.mcp]/list-tools.ts`, `src/routes/[.mcp]/invoke-tool/$tool.ts` — emitted by the `mcpPlugin` Vite plugin from `src/lib/mcp/admin.ts`. To take ownership of one of these, delete its banner comment; the plugin then leaves it alone.
- `src/routeTree.gen.ts` — TanStack Router's generated route tree. Never edit by hand.

## Architecture

### TanStack Start app shell

- File-based routing under `src/routes/`; see `src/routes/README.md` for the full convention table (`$id` dynamic segments, `{-$category}` optional segments, `$` splats, `_layout.tsx`, `__root.tsx`). Do not create Next.js/Remix-style `src/pages/` or `app/` directories.
- `src/routes/__root.tsx` is the single app shell (head tags, fonts, global `<Toaster />`, `<NewBookingDrawer />`) — it wraps every route and must preserve `<Outlet />`.
- `src/router.tsx` creates the router and seeds route context with a `QueryClient` (TanStack Query).
- `vite.config.ts` is intentionally minimal: `@lovable.dev/vite-tanstack-config`'s `defineConfig` already wires up `tanstackStart`, `viteReact`, `tailwindcss`, `tsConfigPaths`, nitro (Cloudflare build target), the dev-only component tagger, `VITE_*` env injection, the `@` path alias, and React/TanStack deduping. Do not re-add any of those plugins manually — duplicate plugins break the build. The `mcpPlugin` (mounting `/mcp`) is the one thing added explicitly here.

### Server entry / error handling

- `src/start.ts` registers global middleware via `createStart`: `attachSupabaseAuth` (client-side function middleware that attaches the Supabase bearer token to server-fn RPCs) and `errorMiddleware` (server-side, catches unhandled errors and renders a fallback error page instead of a raw 500).
- `src/server.ts` wraps the TanStack Start server entry (`@tanstack/react-start/server-entry`) with `normalizeCatastrophicSsrResponse`: h3 sometimes swallows in-handler throws into a generic `{"unhandled":true,...}` 500 JSON body that a plain `try/catch` never sees, so this detects that shape specifically and renders `renderErrorPage()` instead.
- `src/lib/error-capture.ts` listens for global `error`/`unhandledrejection` events and stashes the last error for ~5s so `server.ts` can recover the original stack when h3 has already swallowed it.
- Route-level errors are handled by `errorComponent`/`notFoundComponent` on `src/routes/__root.tsx`, which also reports to `reportLovableError` (`src/lib/lovable-error-reporting.ts`).

### Env / config access patterns

Three distinct patterns are used deliberately (see comments in `src/lib/config.server.ts`):
- **`*.server.ts` module** (e.g. `config.server.ts`, `client.server.ts`) — server-only helpers reused across handlers. On Cloudflare Workers, env binds at *request* time, so reads must happen inside a function, not at module scope.
- **inline `process.env` inside a `createServerFn` handler** — one-off server reads not reused elsewhere.
- **`import.meta.env.VITE_FOO`** — public config readable on both client and server; only ever put non-secret values here (they ship to the browser). Defined in `.env` with a `VITE_` prefix.

`eslint.config.js` bans importing the `server-only` npm package specifically because this codebase uses the `*.server.ts` naming convention instead.

### Supabase integration

- `src/integrations/supabase/client.ts` — browser/SSR-safe client (anon/publishable key), lazily constructed behind a `Proxy` so `import.meta.env`/`process.env` aren't read until first use. Persists session to `localStorage` on the client.
- `src/integrations/supabase/client.server.ts` — **service-role** admin client, bypasses RLS. Server-only; only import at the top level from other `.server.ts` modules (route files and `*.functions.ts` ship to the client bundle, so import it dynamically there: `await import("@/integrations/supabase/client.server")`).
- `src/integrations/supabase/auth-middleware.ts` exports `requireSupabaseAuth`, a server function middleware that validates the `Authorization: Bearer <jwt>` header via `supabase.auth.getClaims`, then injects `{ supabase, userId, claims }` into context. Use this for RLS-scoped server functions.
- `src/integrations/supabase/auth-attacher.ts` exports `attachSupabaseAuth`, the client-side middleware (registered globally in `src/start.ts`) that actually puts the bearer token on outgoing server-fn requests — required for `requireSupabaseAuth` to have anything to check.

### Server functions

`src/lib/api/*.functions.ts` holds `createServerFn` handlers (see `example.functions.ts`) — this is the app's convention for server logic instead of Supabase Edge Functions. `.inputValidator` uses Zod. Only code inside `.handler()` is guaranteed tree-shaken from the client bundle; module-level code in a `.functions.ts` file still ships to the client, so keep server-only imports inside the handler or in a `.server.ts` file.

### MCP server (`/mcp`)

Claro exposes a single authenticated MCP server for AI clients, defined in `src/lib/mcp/admin.ts` via `@lovable.dev/mcp-js`'s `defineMcp`:
- Auth is Supabase OAuth 2.1 (`auth.oauth.issuer`); the issuer **must** be the direct `<project-ref>.supabase.co` host, not the `.lovable.cloud` proxy, because Lovable rewrites `SUPABASE_URL` on publish and `mcp-js` verifies the RFC 8414 issuer match — the project ref is the one Supabase value stable across publishes.
- There is no public/unauthenticated MCP mount.
- Tools live in `src/lib/mcp/tools/*.ts`, each built with `defineTool` (Zod `inputSchema`, `annotations` like `readOnlyHint`, a `handler` returning `{ content, structuredContent }`). Existing tools: `whoami` (echoes the verified caller), `get_app_info` (static app metadata), `estimate_cleaning` (illustrative pricing calc). New tools get added to the `tools: []` array in `admin.ts`.
- The routes under `src/routes/mcp.ts`, `[.well-known]/oauth-protected-resource.ts`, and `[.mcp]/**` are generated wiring for this — see "Generated files" above.

### Client-side data model

There is no backend persistence for the core domain data yet — `src/lib/store.ts` is a hand-rolled client store (module-level state + `useSyncExternalStore`, persisted to `localStorage` under key `claro.store.v1`) holding `properties`, recurring `schedules`, `customBookings`, and per-occurrence overrides (`skipped`, `occurrenceCleaners`). `generateOccurrences()` expands a `RecurringSchedule` into concrete dated occurrences within a rolling window (default 8 weeks), respecting weekly/biweekly/monthly cadence and an optional end date. `src/lib/propertyStatus.ts` derives per-property status badges (`Vacant`/`Occupied`/`Turn Today`/`Check-out Today`) and "next event" strings from this same booking data, shared between the Overview (`/`) and `/properties` pages — keep those two in sync when changing booking/event shapes.

### UI components

`src/components/ui/*` is shadcn/ui (`components.json`: `new-york` style, slate base, Tailwind v4 CSS variables, Lucide icons). `src/components/dashboard/*` holds the app-specific dashboard components (Sidebar, Timeline, BookingsTable, BookingDetail, StatCard, modals/drawers, etc.). Path alias `@/*` maps to `src/*` (configured in both `tsconfig.json` and via `vite-tsconfig-paths`).
